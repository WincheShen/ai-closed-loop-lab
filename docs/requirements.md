# AI 闭环实验室 — 正式需求文档 (v0.2)

> 本文档基于沈经理 2026-04-26 的需求澄清整理，覆盖三个核心模块的功能、边界和接口。

## 0. 项目目标

构建一个 **可持续运营的"AI 实盘 + 内容创作"闭环系统**，目标用户是个人投资者+内容创作者（沈经理），核心要解决：

1. **选股质量**：用 AI 系统化地从全市场筛选，并能持续根据真实交易结果迭代选股逻辑
2. **内容效率**：把每天的交易行为转化为合规、有价值的社交媒体内容，避免封号
3. **流量增长**：通过引流式互动（评论热门文章）持续获得新粉丝
4. **决策辅助**：TradingAgent 作为常驻服务，对任意股票按需提供深度分析报告

## 1. 模块一：股票分析平台 (Stock Analyzer)

### 1.1 定位

**输出端**是给沈经理的"今日推荐买入清单"，**输入端**接收沈经理每日的实际交易记录用于复盘。

### 1.2 功能需求

#### FR-1.1 每日选股流水线
- **数据源**：AKShare 全市场行情、资金流向、龙虎榜、板块涨跌幅
- **触发时机**：每日 15:35（收盘后）自动运行
- **流程**：
  1. 拉取当日全市场数据
  2. 识别**当日热点板块**（涨幅 + 成交量 + 资金净流入综合排序）
  3. 在热门板块中按**选股规则**筛选候选票
  4. 对候选票调用 **TradingAgent 服务** 进行深度分析
  5. **交易员 Agent** 综合评估，输出"最具爆发力" + "最稳妥"两类推荐

#### FR-1.2 选股规则管理
- 必须有独立的**规则配置 + 维护界面**（最低 YAML，未来可视化 UI）
- 规则示例：
  - 量价：5日均量 > 20日均量 × 1.2
  - 技术：站上 MA20，且 MA5 > MA10 > MA20
  - 资金：当日主力净流入 > 5000万
  - 排除：ST、上市不满 60 天、停牌
- 每条规则有**开关、权重、参数**三个维度可调
- 规则版本化，每次修改自动留档（用于 FR-1.3 的归因）

#### FR-1.3 交易结果复盘
- **输入**：沈经理每日通过 wechat/飞书发送的交易记录（买入/卖出 + 股票 + 价位 + 时间）
- **复盘维度**：
  - 选出的股票实际收益 vs 预测收益
  - 哪些规则命中盈利股，哪些规则命中亏损股
  - 错误归因：选股逻辑失效 / 交易规则失效 / 市场系统性风险
- **输出**：
  - 每周末自动生成《选股策略优化建议》报告
  - 给出具体可执行的规则参数调整建议（例如："建议将 MA20 站上权重从 0.3 调到 0.2"）

#### FR-1.4 交易员推荐逻辑
- 不是简单的分数排序，而是结合：
  - 热门板块归属强度
  - TradingAgent 的多空辩论结果
  - 当前持仓集中度（避免同一板块过度配置）
- 输出格式：
  - **激进推荐**（爆发力优先）：1-3 只
  - **稳健推荐**（确定性优先）：1-3 只
  - 每只股票必须给出：买入价区间、止损位、目标价、推荐理由（≤200字）

### 1.3 非功能需求

- 选股结果必须**当晚 18:00 前**产出（沈经理晚饭后查看）
- TradingAgent 调用必须走 Cache（见 §3），避免重复深度分析

## 2. 模块二：Social Media Automation

### 2.1 定位

**不主动搜热点**，所有内容创作都基于股票分析平台输出的"今日热门板块/股票"信号。专注于**合规创作 + 引流互动 + 数据反哺**。

### 2.2 功能需求

#### FR-2.1 数据驱动的选题（不主动搜热点）
- 选题输入**只来自**股票分析平台输出的：
  - 当日热门板块
  - 推荐买入股票
  - 沈经理的实际交易记录（来自 webhook）
- 禁止：自行搜全网热点、自行决定选题

#### FR-2.2 主题相关的内容研究
- 收到选题后，从小红书搜索**最近 1-2 天**该主题的爆款文章（点赞 / 评论 / 收藏排序）
- 分析爆款文章的：
  - 标题套路
  - 核心观点
  - 配图风格
  - 用户互动模式
- **输出**：研究报告 → 输入给创作引擎

#### FR-2.3 交易信息接收 + 合规处理
- **接收通道**：wechat / 飞书机器人 webhook（沈经理手动发）
- **接收内容**：图片（持仓截图）+ 文字（买卖点 + 理由）
- **合规处理**（防封号）：
  - 图片：自动模糊股票代码、股票名称、具体仓位、账户余额
  - 文字：替换敏感词（"买入" → "上车"、"建仓" → "关注"），具体价格区间化（"19.8" → "20元附近"）
  - 不能出现"老师推荐"、"必涨"、"内幕"等违规词
- **输出**：脱敏后的图片 + 改写后的文字 → 输入给创作引擎

#### FR-2.4 内容创作引擎
- 综合输入：
  - 股票分析平台的板块/股票分析结论（提供逻辑支撑）
  - 沈经理的脱敏交易信息（提供真实感）
  - 小红书爆款研究报告（提供文风模板）
- 输出小红书笔记：标题 + 正文 + 图片（含脱敏图）+ 标签

#### FR-2.5 二次加工模式（人工选题）
- 沈经理手工提供主题/草稿
- Agent 优化文案 + 生成配图 + 发布
- 与 FR-2.4 共用底层创作引擎，差异在选题来源

#### FR-2.6 引流模块（评论引流）
- **每日热门文章扫描**：根据沈经理的小红书 topic（如"A股投资"、"沪深300"）搜索当日最热文章
- **针对性评论生成**：基于文章内容，写**有价值的、个性化**评论（不是广告）
- **评论发布**：通过 `xhs_cli` 发布
- **频次控制**：每日 ≤ 20 条，避免被风控
- **黑名单**：不评论政治敏感、纯营销账号、已封号账号

#### FR-2.7 自动回复评论
- **监听**：自有发布笔记下的新评论
- **分级回复**：
  - 高质量提问（含投资观点）→ AI 详细回复
  - 简单赞美（"厉害"、"学到了"）→ 简短感谢回复
  - 引战 / 嘲讽 → 不回复 + 标记
  - 营销广告 → 不回复 + 删除（如有权限）
- **频次控制**：单日单账号 ≤ 50 条回复

#### FR-2.8 运营数据收集 + 自我优化
- **每日收集**：
  - 发布笔记的曝光、点击、点赞、收藏、评论数
  - 引流评论的点赞数、被回复数
  - 涨粉数、取关数
- **每周自我优化**：
  - 哪类标题点击率最高
  - 哪类配图点赞率最高
  - 哪类话题涨粉最快
  - 输出《下周内容策略调整建议》

#### FR-2.9 评论价值反哺（连接到模块一）
- 自有笔记下的高质量评论（如"这个票最近主力出货明显"）
- AI 提炼成"外部观察"事实卡片
- 反向输入到股票分析平台的选股流程，作为补充维度

### 2.3 非功能需求

- 所有发布默认走 `review_mode: review`（人工审核），稳定后可切到 `auto`
- 每个账号独立浏览器 Profile + 代理 IP（已有，沿用 `xiaohongshu-skills`）

## 3. 模块三：TradingAgent 服务

### 3.1 定位

**长期运行的 HTTP 服务**，对外提供单股深度分析能力。被股票分析平台、Social Media、外部用户三方调用。

### 3.2 功能需求

#### FR-3.1 服务化（不再是一次性脚本）
- HTTP API（FastAPI）：
  - `POST /analyze` — 分析单股，返回完整报告
  - `GET /report/{symbol}` — 获取最新缓存报告
  - `GET /health` — 健康检查
  - `GET /stats` — 调用统计、缓存命中率
- 启动方式：`uvicorn` 守护进程 / Docker 容器
- 单机部署（数据隔离要求）

#### FR-3.2 缓存层
- **缓存键**：`(symbol, date)` 维度
- **失效条件**：
  - 报告中标注的"再评估价格"被触发（股价超出预设区间）
  - 报告超过 N 天（默认 3 天）
  - 该股票发布重大公告 / 新增龙虎榜
- **缓存命中**：
  - 直接返回上次报告
  - **必须**在报告头部注明：`📅 评估日期: 2026-04-25 (缓存命中)`
- **缓存存储**：SQLite + 报告全文（JSON / Markdown）

#### FR-3.3 知识星球自动同步
- 每生成一份新报告（非缓存命中），自动：
  1. 格式化为知识星球友好格式（Markdown + 配图）
  2. 调用知识星球 API（或浏览器自动化，待评估）发布
  3. 记录发布状态 + URL，回写到报告元数据
- **重要**：缓存命中的报告**不重复发布**

#### FR-3.4 报告内容规范
- 必须包含：
  - 当前价格、技术形态摘要
  - 基本面摘要（行业、市值、PE/PB、ROE）
  - 多空双方观点（来自 TradingAgents 的辩论结果）
  - **再评估价格区间**（用于 Cache 失效判定）
  - 风险提示
- 评估日期戳必须**清晰可见**

### 3.3 非功能需求

- 单股分析延迟 ≤ 90 秒（深度分析）/ ≤ 1 秒（缓存命中）
- 服务可用性 ≥ 99%（白天交易时段）
- API 限流：单 IP ≤ 10 req/min，避免误用导致 LLM 费用暴增

## 5. 优先级排序（建议实现顺序）

| 优先级 | 模块 | 功能 | 备注 |
|:------:|------|------|------|
| **P0** | 模块3 | TradingAgent 服务化 + Cache | 其他模块的基础能力 |
| **P0** | 模块1 | 选股规则管理 + 每日选股流水线 | 整个闭环的入口 |
| **P0** | 模块2 | Webhook 接收 + 交易信息合规处理 + 创作引擎 | 已有部分基础 |
| **P1** | 模块3 | 知识星球同步 | 需要调研 API 可行性 |
| **P1** | 模块1 | 交易结果复盘 + 优化建议 | 依赖一段时间的真实数据 |
| **P1** | 模块2 | 引流评论模块 | 需先解决风控问题 |
| **P2** | 模块2 | 自动回复评论 | 风险较高，最后做 |
| **P2** | 模块2 | 运营数据收集 + 自我优化 | 需有足够运营数据后启动 |
| **P2** | 模块1 | 评论价值反哺 | 锦上添花 |

## 6. 风险与边界

### 6.1 合规风险（最高优先级）
- **绝不**发布"必涨"、"老师"、"内幕"等违规词
- **绝不**暴露具体仓位、账户金额
- 引流评论**必须**有真实价值，禁止"机器人式刷评论"

### 6.2 资金安全
- 所有交易信号默认 `mock` 模式
- 实盘上线需要：1）模拟盘连续 30 天稳定 2）人工最终确认机制

### 6.3 技术债务
- 已有的 `tradingAgents_neo` 是单次脚本，需要重构成服务（最大改动）
- 已有的 `Social-media-automation` 已是 LangGraph 架构，扩展即可
